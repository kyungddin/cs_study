# Chap 1: 가상화

## System Configuration
- 하드웨어 - 커널 - 운영체제 - 셸 - 애플리케이션
    - 셸은 OS와 사용자 사이에서 다리 역할을 함. 즉 운영체제와 상호작용을 돕는 친구.
- 네임스페이스: 프로세스를 실행할 때 시스템 리소스를 분리해서 실행할 수 있도록 도와주는 기능
    - 리눅스 네임스페이스 ex: pid, net, mnt 등등..


## 도커 기초 지식
- 도커: 컨테이너라고 부르는 운영체제 수준의 가상화 방식으로 소프트웨어를 배포하는 방식을 사용하는 PAAS 제품
    - PAAS: Platform As A Service
    - 즉 App/데이터 단계만 사용자가 관리하는 서비스
- Hierarchy: 하드웨어 - 운영체제 - 도커 - 컨테이너들 (with 애플리케이션, 바이너리/라이브러리)


## 가상화 개념
- 호스트 가상화: 하드웨어 - 호스트 OS - 하이퍼바이저 - VM들(with 게스트 OS, 바이너리/라이브러리, App)
- 하이퍼바이저 가상화: 하드웨어 - 하이퍼바이저 - VM들
- 컨테이너 가상화: 하드웨어 - OS - 도커 - 컨테이너들
    - 컨테이너 가상화의 문제점
        - 게스트 OS를 포함하지 않는 것처럼 보이는 이유는 리눅스 커널을 빌려쓰기에, 호스트 OS가 리눅스가 아니라면 리눅스 커널과 리눅스 가상 머신이 필요하다
        - 그럼에도 게스트 OS를 올리는 호스트 가상화보다는 가볍다..! (Guest OS의 커널을 거치지 않으므로)
    - 컨테이너 가상화의 경우 호스트 OS의 아키텍처를 따라서 리눅스 커널의 아키텍처가 결정된다!


## 커널 공유
- Docker는 무조건 Linux 커널을 사용하고, 모든 컨테이너는 이를 공유함
- 하지만 만약 호스트 OS가 윈도우고, 컨테이너 이미지가 윈도우 기반이면 윈도우 커널 사용
- 단, 아무리 커널을 공유해서 가볍고 빠르다고 하더라도 아키텍처가 다르다면 커널 공유가 사실상 불가능


## Namespace vs Cgroup (좀 더 공부하기)
```
| 구분        | Namespace     | cgroup       |
| --------- | ------------- | ------------ |
| 목적        | 격리            | 제한/관리        |
| 대상        | 보이는 환경        | 자원 사용량       |
| 느낌        | “서로 다른 컴퓨터처럼” | “CPU/RAM 배급” |
| Docker 역할 | 컨테이너 분리       | 컨테이너 성능 제어   |
```
- Namespace는 한 OS 내에서 프로세스를 독립시키는 것
- cgroup은 한 프로세스 내에서 자원을 독립시키는 것, 리눅스 커널?
- 둘 다 docker 컨테이너의 독립을 위해 필요한 요소임


# Chap 2: 도커 Components

## 도커 엔진
- 도커 엔진 구성 3요소
    - 데몬(상주 프로세스), API, CLI클라이언트
    - CLI를 통해 우리가 docker에 명령을 전달하면, API를 거쳐 데몬에 명령이 전달된다


## 도커 컴포즈
- Docker 컨테이너가 증가함에 따라 이를 명령어 하나로 동시에 조작하게 해주는 기능
- YAML 파일 정의에 의해 동시에 관리


## 도커 허브
- 클라우드에 이미지(컨테이너 모형)을 공개하는 곳
- 수 많은 리포지토리에 여러 이미지가 등록되어 있음


# Chap 4: 도커 Basic

## 컨테이너
- 컨테이너는 어떤 명령어 하나를 실행하기 위해 호스트머신에 작성된 영역

## 이미지
- 컨테이너 실행에 필요한 패키지
- 다수의 레이어로 구성
- 각 레이어는 tar 아카이브 파일
- 이미지를 기반으로 컨테이너가 실행됨
    - 굳이 비유를 하자면 이미지는 프로그램이고, 컨테이너는 프로세스라고 한다
